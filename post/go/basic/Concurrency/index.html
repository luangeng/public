Go 并发

Posted @ 2018-09-09 · 2275 Words · Tags: Go · Hits:

Go使用goroutines实现并发. 简单来说goroutines就像线程.

Gorutines独立执行并且共享内存.

由于goroutines可以写入相同的内存，因此多个goroutine写入相同的全局变量时需要格外小心。

为了协调goroutine之间的工作，Go提供了channel,即线程安全队列。

goroutine示例

单channel，单goroutine，一次写入，一次读取。

func main() {
    // 创建string型channel
    ch := make(chan string)

    // 启动一个匿名goroutine
    go func() {
        time.Sleep(time.Second)
        // 发送"Hello World"到channel
        ch <- "Hello World"
    }()
    // 从channel读取
    msg, ok := <-ch
    fmt.Printf("msg='%s', ok='%v'\n", msg, ok)
}

msg='Hello World’, ok='true’

这个channel ch是无缓冲的或者说是同步的。

time.Sleep在这里说明main()方法将在channel上等待，这意味着goroutine执行的函数需要一定的时间才能通过channel发送值,接收操作符<-ch将阻止main()方法执行。

如果不是，则goroutine将在main()退出时被杀死，并且没有时间发送其值。

创建goroutines

任何函数都可以通过使用关键字go来作为goroutine调用：

func mult(x, y int) {
    fmt.Printf("%d * %d = %d\n", x, y, x*y)
}

go mult(1, 2) // 第一次执行,非阻塞
go mult(3, 4) // 第二次执行,非阻塞

1 * 2 = 2
3 * 4 = 12

请注意，该函数的返回值被忽略。

等待goroutine完成

Go程序在主函数结束时结束，因此通常的做法是等待所有goroutine完成。

常见的解决方案是使用sync.WaitGroup对象。

var wg sync.WaitGroup // 1

func routine(i int) {
    defer wg.Done() // 3
    fmt.Printf("routine %v finished\n", i)
}

func main() {
    wg.Add(10) // 2
    for i := 0; i < 10; i++ {
        go routine(i) // *
    }
    wg.Wait() // 4
    fmt.Println("main finished")
}

routine 3 finished
routine 9 finished
routine 0 finished
routine 4 finished
routine 5 finished
routine 6 finished
routine 7 finished
routine 8 finished
routine 1 finished
routine 2 finished
main finished

执行顺序中的WaitGroup用法：

声明全局变量。 使它全局化是使其对所有功能和方法可见的最简单方法。 增加柜台。 这必须在主goroutine中完成，因为由于内存模型的保证，不能保证新启动的goroutine将在4之前执行。 减少计数器。 这必须在goroutine的出口处完成。 通过使用延迟调用，我们确保无论函数如何结束，只要函数结束，都将调用它。 等待计数器达到0。必须在主goroutine中执行此操作，以防止程序在所有goroutine完成之前退出。 在开始新的goroutine之前先评估参数。

因此，有必要在wg.Add(10)之前显式定义其值，以免出现恐慌的代码不会增加计数器。 向WaitGroup添加10个项目，因此它将在wg.Wait之前将等待10个项目。Wait将控件返回给main()goroutine。 在这里，i的值在for循环中定义。

用semaphore限制并发

当执行CPU密集型任务(如调整图像大小)时，创建比可用CPU多的goroutine毫无意义。

事情不会进展得更快，由于额外的簿记和在goroutine之间进行切换，您甚至可能会失去性能。

限制并发性(即同时启动的gorutine的数量)的一种方法是使用semaphore。

您可以进入(获取)semaphore并离开(释放)semaphore。

semaphore具有固定的容量。如果超出了信号量的容量，获取semaphore将阻塞，直到有gorutine释放semaphore。

缓冲channel是自然的semaphore。

这是一个使用channel作为semaphore来限制在任何给定时间激活的goroutine数量的示例：

var (
    semaphoreSize = 4

    mu                 sync.Mutex
    totalTasks         int
    curConcurrentTasks int
    maxConcurrentTasks int
)

func timeConsumingTask() {
    mu.Lock()
    totalTasks++
    curConcurrentTasks++
    if curConcurrentTasks > maxConcurrentTasks {
        maxConcurrentTasks = curConcurrentTasks
    }
    mu.Unlock()

    // in real system this would be a CPU intensive operation
    time.Sleep(10 * time.Millisecond)

    mu.Lock()
    curConcurrentTasks--
    mu.Unlock()
}

func main() {
    sem := make(chan struct{}, semaphoreSize)
    var wg sync.WaitGroup
    for i := 0; i < 32; i++ {
        // acquire semaphore
        sem <- struct{}{}
        wg.Add(1)

        go func() {
            timeConsumingTask()
            // release semaphore
            <-sem
            wg.Done()
        }()
    }

    // wait for all task to finish
    wg.Wait()

    fmt.Printf("total tasks         : %d\n", totalTasks)
    fmt.Printf("max concurrent tasks: %d\n", maxConcurrentTasks)
}

total tasks : 32
max concurrent tasks: 4

我们使用等待goroutine完成时描述的技术来等待所有任务完成。

通常，正确的并发任务数量等于可以通过runtime.NumCPU()获得的CPU数量。

下面是一个使用goroutine的工作程序池并将其工作与channel进行协调的示例：

var wg sync.WaitGroup

func sqrtWorker(chIn chan int, chOut chan int) {
    fmt.Printf("sqrtWorker started\n")
    for i := range chIn {
        sqrt := i * i
        chOut <- sqrt
    }
    fmt.Printf("sqrtWorker finished\n")
    wg.Done()
}

func main() {
    chIn := make(chan int)
    chOut := make(chan int)
    for i := 0; i < 2; i++ {
        wg.Add(1)
        go sqrtWorker(chIn, chOut)
    }

    go func() {
        chIn <- 2
        chIn <- 4
        close(chIn)
    }()

    go func() {
        wg.Wait()
        close(chOut)
    }()

    for sqrt := range chOut {
        fmt.Printf("Got sqrt: %d\n", sqrt)
    }
}

sqrtWorker started
Got sqrt: 4
sqrtWorker started
sqrtWorker finished
sqrtWorker finished
Got sqrt: 16

这里有很多东西要解释。

我们使用go sqrtWorker(chIn，chOut)启动2个工作程序。

每个sqrtWorker函数均独立运行，并与所有其他代码并行运行。

我们使用int值的单个channel来对工作项进行排队，这些工作项将由工作程序goroutine在channel上使用<-send操作来处理。

在大多数情况下，从多个goroutine访问同一变量并不安全, channel和sync.WaitGroup例外。

Worker goroutines sqrtWorker使用range从channel中获取值。

We don’t know which worker will pick any given value.

Worker’s for loop terminates when chIn is closed with close(chIn) and worker goroutine terminates.

To pass results of worker goroutines back to the caller we use another channel.

In this example we use unbuffered channels which only have capacity for one item at a time. For that reason we launch another goroutine to fill chIn. Otherwise we would risk dead-lock.

To shutdown workers we close the chIn.

We then wait for results created by workers by iterating on chOut.

There’s one more complication. Unless we close chOut, the for sqrt := range chOut loop will wait forever.

To stop the loop, we need to close(chOut) but when to do it?

We can’t do it in sqrtWorker because there are many of them and calling close on an already closed channel will panic.

sync.WaitGroup is a thread-safe counter that can be incremented / decremented and allows for waiting until the counter reaches 0.

Before we launch the worker, we increment wg counter.

Just before terminating, the worker decrements wg counter.

We then wg.Wait() for the counter to reach 0, which indicates that all workers have finished and it’s now safe to close the output channel.

It has to happen in its own goroutine to avoid blocking.

Why go to all this trouble to calculate a simple value?

This is just an example. In a real programs worker goroutine would perform longer jobs like downloading a file from the internet or resizing a large image.
