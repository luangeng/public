Go 错误处理

Posted @ 2018-09-09 · 2449 Words · Tags: Go · Hits:

与C#或Python等语言不同，Go通过返回错误值而不是抛出异常来表示程序故障。

Go同时提供了panic和recover，但很少使用。

错误是值，就像整数或字符串一样。

类型错误是实现Error()字符串方法的内置接口。

如果表示成功，请返回nil。

要创建错误，可以使用标准库函数errors.New(msg字符串)或fmt.Errorf(格式字符串，args … interface {})。

你还可以通过定义实现Error()字符串方法的结构来返回自定义错误类型。

如果函数返回错误，则应始终为最后一个返回值.

始终检查返回的错误并妥善处理而不是忽略它。

返回错误

为了指示一个错误,函数应该返回一个错误值.函数没有错误时,返回nil.

// somewhat inefficient way of appending a string at the end of a file
func appendToFile(path string, s string) error {
    d, err := ioutil.ReadFile(path)
  if err != nil {
        return err
  }
    d = append(d, []byte(s)...)
  return ioutil.WriteFile(path, d, 0644)
}

如果函数有多个返回值时,error始终应该是最后一个:

func sqrt(n int) (int, error) {
    if n < 0 {
        return 0, fmt.Errorf("n should be >= 0, is %d", n)
    }
    return n*n, nil
}

基本错误处理:

func sqrt(n float64) (float64, error) {
    if n < 0 {
        return 0, fmt.Errorf("invalid argument '%f', must be >= 0", n)
    }
    return math.Sqrt(n), nil
}

func printSqrt(n float64) {
    if res, err := sqrt(n); err == nil {
        fmt.Printf("sqrt of %f is %f\n", n, res)
    } else {
        fmt.Printf("sqrt of %f returned error '%s'\n", n, err)
    }

}

func main() {
    printSqrt(16)
    printSqrt(-16)
}

sqrt of 16.000000 is 4.000000
sqrt of -16.000000 returned error ‘invalid argument ‘-16.000000’, must be >= 0’

Pitfalls

如果你使用的是自定义错误类型，请不要返回底层类型，而请务必返回错误类型。

type CustomError struct {
}

func (e *CustomError) Error() string {
    return "this is a custom error"
}

func returnsCustomError(pleaseFail bool) *CustomError {
    if pleaseFail {
        return &CustomError{}
    }
    return nil
}

func caller() {
    var err error
    if err == nil {
        fmt.Printf("err is nil, as expected\n");
  }
    err = returnsCustomError(false))
    if err != nil {
        fmt.Printf("returnCustomError() failed with '%s'\n", err)
  }
}

出乎意料的是，即使返回null，它看起来也像returnCustomError()返回了错误。

这是nil接口陷阱的一种变体。

未初始化的err值为nil，因为它是接口类型的零值。

returnCustomError()返回CustomError结构的nil值。

此nil值分配给err变量。 这等效于：

var err error
err = (*CustomError)(nil)

此时，err不再等于接口类型的零值。 它的值为* CustomError，其值为nil。

因此，err不再等于nil。

换句话说：

var err1 error
var err2 error = nil
var err3 error = (*CustomError)(nil)
if err1 == nil && err2 == nil {
    fmt.Printf("unitialized interface value and inteface value assigned nil are both zero values of interface type and therefore equal to 'nil'\n")
}
if err3 != nil {
    fmt.Printf("interface value assigned to a struct type whose value is nil is not zero value of interface type and therefore not equal to 'nil'\n")
}

创建标准错误值

有几种创建错误值的方法:

  • errors.New 最简单的方式, errors.New(msg string)从字符串创建错误

return errors.New("error created with errors.New")

  • fmt.Errorf 用来打印复杂的错误, fmt.Errorf(format string, args …interface{}) 可接受格式化string和参数

return fmt.Errorf("error created with %s", "fmt.Errorf") 这是errors.New(fmt.Sprintf(…))的简写

全局错误变量

// ErrGlobal exported so that others can compare returned error value with this variable
var ErrGlobal = errors.New("global error variable")

return ErrGlobal

有时希望错误具有标识，以便调用者可以测试返回的错误是否是此特定错误。 则可以通过在上面的示例中将全局变量声明为ErrGlobal来实现。

标准库中此类错误的一个示例是io.EOF，尽管通常此类错误的命名约定为Err *。

你还可以创建自定义错误类型。

自定义错误

在Go中，错误类型是带有单个方法Error()字符串的内置接口。

所有管理接口的规则均适用于错误类型。

其中，任何实现错误接口的类型都可以用作错误类型的值。

// MyError is a custom error type
type MyError struct {
    msg string
}

func (e *MyError) Error() string {
    return e.msg
}

var err error = &MyError{msg: "This is custom error type"}
fmt.Printf("err: %s\n", err)

err: This is custom error type

除了创建错误的内置方法之外，为什么还要使用自定义类型？

潜在的用例是：

包装现有错误并添加更多信息，例如调用堆栈 向错误添加结构化信息。 处理该错误的代码可以使用类型断言或类型切换检查自定义类型，并访问此信息

处理错误

在Go中，可以从函数调用中返回错误。 约定是，如果方法可能失败，则最后返回的参数是错误。

func DoAndReturnSomething() (string, error) {
    if os.Getenv("ERROR") == "1" {
        return "", errors.New("The method failed")
    }

    // The method succeeded.
    return "Success!", nil
}

你使用多个变量分配来检查方法是否失败。

result, err := DoAndReturnSomething()
if err != nil {
    panic(err)
}

// This is executed only if the method didn't return an error
fmt.Println(result)

如果你对该错误不感兴趣，则可以通过将其分配给_来忽略它。

result, _ := DoAndReturnSomething()
fmt.Println(result)

当然，忽略错误可能会带来严重的影响。 因此，通常不建议这样做。

如果你有多个方法调用，并且链中的一个或多个方法可能返回错误，则应将错误传播到可以处理该错误的第一级。

func Foo() error {
    return errors.New("I failed!")
}

func Bar() (string, error) {
    err := Foo()
    if err != nil {
        return "", err
    }

    return "I succeeded", nil
}

func Baz() (string, string, error) {
    res, err := Bar()
    if err != nil {
        return "", "", err
    }

    return "Foo", "Bar", nil
}

在短程序中处理错误

将错误传播给调用者是很乏味的, 最终会看到许多类似以下内容的行：

r, err := os.Open("my file")
if err != nil {
    return err
}

这种错误的处理对于编写健壮的软件至关重要。

有时编写了较短的cmd-line程序，而这些检查是没有必要的。

这时可以使用辅助函数PanicIfErr(err error，args … interface {})来简化错误处理：

r, err := os.Open("my file")
PanicIfErr(err)

此类助手的实现：

// FmtArgs formats args as a string. First argument should be format string
// and the rest are arguments to the format
func FmtArgs(args ...interface{}) string {
    if len(args) == 0 {
        return ""
    }
    format := args[0].(string)
    if len(args) == 1 {
        return format
    }
    return fmt.Sprintf(format, args[1:]...)
}

func panicWithMsg(defaultMsg string, args ...interface{}) {
    s := FmtArgs(args...)
    if s == "" {
        s = defaultMsg
    }
    fmt.Printf("%s\n", s)
    panic(s)
}

// PanicIf panics if cond is true
func PanicIf(cond bool, args ...interface{}) {
    if !cond {
        return
    }
    panicWithMsg("PanicIf: condition failed", args...)
}

// PanicIfErr panics if err is not nil
func PanicIfErr(err error, args ...interface{}) {
    if err == nil {
        return
    }
    panicWithMsg(err.Error(), args...)
}

func main() {
    PanicIfErr(nil)                              // nothing happens
    PanicIfErr(errors.New("there was an error")) // will panic
}

将调用堆栈添加到错误消息

如果你来自Java，C＃或Python之类的语言，则你可能会习惯于这样的事实，即异常包括用于创建异常的位置的调用堆栈。

收集调用栈非常昂贵，并且Go不会将调用栈添加到错误中。

调用堆栈在调试中很有用。 如果你可以接受额外费用，则可以使用github.com/pkg/errors包来增加callstack的错误。

func wrappedError() error {
    err := errors.New("Original error")
    // create a new error value which wraps original err and
    // adds calls tack
    return pkgerrors.WithStack(err)
}

func main() {
    // %+v prints original error add callstack
    fmt.Printf("err: %+v\n\n", wrappedError())

    // errors created with pkg/errors include callstack by default
    fmt.Printf("err: %+v\n", pkgerrors.New("error created with pkg/errors"))
}

可见，调用栈的顶部包括Go运行时代码。

编写良好的错误消息

良好的错误消息有助于调试问题。

在生产部署中，将记录错误消息。 当出现问题时，日志文件是调试线索的主要来源。

在创建错误消息时，包括尽可能多的有用信息。

不好的错误消息：

func openLog(path string) (io.Reader, error) {
    f, err := os.Open(path)
    if err != nil {
        return nil, errors.New("failed to open log file")
    }
}

好的错误消息:

func openLog(path string) (io.Reader, error) {
    f, err := os.Open(path)
    if err != nil {
        return nil, errors.New("openLog('%s'), os.Open() failed with '%s'", path, err)
    }
}

好的错误信息包含更多调试线索：哪个函数生成了错误,我们正试图打开的文件的路径,来自os.Open()的错误消息，以便判断是否是因为文件没有存在，或者没有访问权限等。
